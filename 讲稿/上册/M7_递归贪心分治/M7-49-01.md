# 第49章 第1课时：分治思想——分、治、合

**时长**：15-20分钟 | **对应教材**：第49章 49.1-49.2

## 🎯 本课时目标
- 掌握分治三步骤：分、治、合
- 区分分治和减治
- 用分治视角重新审视汉诺塔和归并排序

## 📝 讲稿正文

前面的贪心是"一条路走到黑"，今天学的分治恰恰相反——**把大问题切成好几个小问题，各自解决后再拼起来**。

同学们，你们其实早就用过"分治"了，只是没意识到。老师让你统计全班 50 份作业的平均分，你怎么做？一个人算 50 份太慢——你把这 50 份分给 5 个小组长，每人算 10 份，然后把结果汇总。这就是分治。

分治有三步：**分**（Divide）——把原问题切分成若干个子问题；**治**（Conquer）——递归解决每个子问题；**合**（Combine）——把子问题的解合并成原问题的解。

回忆一下汉诺塔。n 个盘子的汉诺塔怎么分的？把上面 n-1 个盘子当成一个子问题——先移到辅助柱（子问题 1），然后最底下盘子直接移（一步），再把 n-1 个盘子从辅助移到目标（子问题 2）。这是分治吗？是！它切成了两个 n-1 规模的子问题，而且最终要"合"——两个子问题的步骤合在一起才是完整方案。

再回忆归并排序。整个数组切成左半和右半，分别排序，然后合并两个有序数组。这是标准的分治。

这里有一个容易混淆的地方——**分治 vs 减治**。它们有什么区别？

减治是"每次只产生一个子问题"，典型例子是阶乘：n! = n × (n-1)!，只有一个递归调用。二分查找也是减治——每次只在左半或右半里找，不是两边都找。

分治是"每次产生两个或更多子问题，而且需要合并结果"。汉诺塔有两个子问题，归并排序有两个子问题——这才是分治。

简单的判断方法：看递归调用次数。如果函数体里只有一个递归调用（self-call once），是减治；如果有多个递归调用且需要合并结果，是分治。

为什么会有人把二分查找称为"分治"？因为二分也是"把问题砍一半"。但严谨地说，二分只砍一条分支，不合并——是减治。不过这种区分不太重要，重要的是理解每次砍掉多少、需不需要合并。

我们用代码来统一分治的框架：

```cpp
返回类型 solve(问题) {
    if (问题足够小) return 直接求解;        // 边界
    拆分成子问题1, 子问题2, ...;          // 分
    结果1 = solve(子问题1);               // 治
    结果2 = solve(子问题2);
    return 合并(结果1, 结果2, ...);       // 合
}
```

三步清清楚楚。以后你们写分治代码，就按这个模板来——先想想分成几份、怎么递归、怎么合并。合并这一步往往是最见功力的地方。

## 🖥️ 板书/屏幕要点
- 分治三步骤：分（Divide）→ 治（Conquer）→ 合（Combine）
- 分治 vs 减治：子问题个数（≥2 vs 1）、是否需要合并
- 例子：汉诺塔（分治）、归并排序（分治）、阶乘（减治）、二分查找（减治）

## ✏️ 随堂练习

2. 写出分治的伪代码框架，标注每一步对应的部分
3. 判断：最大公约数（辗转相除法）是分治还是减治？

## 📌 小结
分治三步骤是本章的灵魂——分、治、合。区分分治和减治的关键是子问题数量和是否需要合并。下节课我们深入归并排序，看看分治在排序中是如何发挥威力的。
