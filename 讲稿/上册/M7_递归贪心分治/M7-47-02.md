## 📝 讲稿正文

上节课我们说 check 函数是二分答案的灵魂。今天把它彻底讲透。

二分答案无非两个方向：**求最大的可行值**，或者**求最小的可行值**。方向不同，模板略有差异。

先看**求最大值**的模板。场景：H 越小越容易满足，你要找满足条件的"最大的 H"。砍树问题就是这样。

```cpp
int l = 最小可能值, r = 最大可能值, ans = l;
while (l <= r) {
    int mid = l + (r - l) / 2;
    if (check(mid)) {
        ans = mid;       // 记录当前可行答案
        l = mid + 1;     // 试试更大的
    } else {
        r = mid - 1;
    }
}
```

关键在第 6-7 行：可行的时候，记录答案，然后**往大了试**（l = mid + 1）。不行的时候往小了缩。ans 初始化为"最小的可能值"——也就是最保守的可行答案。万一 check 全部失败，ans 保持最小可能值，表示"连最宽松的条件都不满足"。

再看**求最小值**的模板。场景：某种条件越紧越难满足，你要找满足条件的"最小的某个值"。比如："最小化最大段和"。

```cpp
int l = 最小可能值, r = 最大可能值, ans = r;
while (l <= r) {
    int mid = l + (r - l) / 2;
    if (check(mid)) {
        ans = mid;       // 记录当前可行答案
        r = mid - 1;     // 试试更小的
    } else {
        l = mid + 1;
    }
}
```

这次反过来：可行的时候，记录答案，然后**往小了试**（r = mid - 1）。ans 初始化为最大可能值——最宽松的可行答案。

两个模板的区别就一句话：**求最大，可行时 l=mid+1；求最小，可行时 r=mid-1**。

check 函数怎么写？这完全取决于题目。但有几条基本原则：

第一，check 必须高效。二分答案的总复杂度 = 二分次数 × check 复杂度。二分次数是 O(log 范围)，所以 check 通常是 O(n) 或 O(n log n)。

第二，check 必须正确映射"可行"的定义。什么算可行？要仔细读题。比如砍树："砍下的木材总量 ≥ M"是可行。不要写反。

第三，注意数据范围。check 里的中间结果经常需要 long long。

第四，注意 check 的单调性是否成立。二分答案写错了，往往不是你二分框架有 bug，而是 check 函数没有单调性——mid 大时可行，mid 小时反而不可行，那就不是"二分"能解决的问题。

给大家一个思考框架：拿到二分答案的题，先问自己三个问题。一：答案的范围是多少？二：check 函数怎么定义？三：check 的结果随答案单调递增还是递减？回答清楚这三个问题，代码就水到渠成了。

## 💬 互动提问
1. "求最大值时，ans 初始化为 l；求最小值时，ans 初始化为 r。为什么？如果反过来会怎样？"
2. "如果 check(mid) 的单调性是'mid 越小越容易可行'，二分方向应该怎么写？"
3. "check 函数里如果用了排序或贪心遍历，复杂度可能是 O(n log n)。这会影响二分答案的适用性吗？"

## 🖥️ 板书/屏幕要点
- 求最大值：可行 → ans=mid, l=mid+1；ans 初始为 l
- 求最小值：可行 → ans=mid, r=mid-1；ans 初始为 r
- check 四原则：高效、正确映射"可行"、注意类型、确保单调性

## ✏️ 随堂练习
1. 给砍树问题写两个版本：求最大 H 和求最小 H（如果题目改为 H 越大越好但至少满足 M），对比二分方向
2. 设计一个 check 反例：mid 增大时 check 有时 true 有时 false 却还能 true，说明它不满足单调性

## 📌 小结
二分答案的两大模板要刻在脑子里——求大往大试，求小往小试。check 函数是核心竞争力，需要仔细设计。下节课我们用"跳石头"这个经典问题来练习"最小值最大化"的范式。
