## 📝 讲稿正文

前面几节课我们分别学了阶乘和斐波那契。今天做两件事：第一，系统对比递归和递推；第二，再学几个实用的递归例子，帮你建立递归"肌肉记忆"。

先来看这张对比表（板书）：

| | 递归 | 递推（循环） |
|------|------|------|
| 写法 | 简洁优雅，贴近数学定义 | 用循环，稍显啰嗦 |
| 思路 | 自顶向下（大→小→更小） | 自底向上（小→大→目标） |
| 效率 | 可能重复计算，占栈空间 | 通常更快，不占栈 |
| 适用 | 天生递归的问题（树、分治） | 线性递推（阶乘、斐波那契） |

简单说：**递归胜在表达力，递推胜在执行力。**

数学定义说 n! = n × (n-1)!，你用递归翻译，几乎一个字不改。但递推要"倒过来想"——从 1 开始乘起。两种思维，各有优势。

**那什么时候该用递归，什么时候该用递推？**

如果一个问题的递归深度不大（几十几百层），而且没有大量重复计算——用递归，代码清楚。
如果深度可能很大（几千几万层），或者有大量重复计算——用递推，或加记忆化。

在算法竞赛里，常见策略是：**先用递归写出正确版本，测试通过后，如果性能不够再改写递推。**

好，现在学两个新例子。

**例子一：辗转相除法求最大公约数（GCD）**（板书）：

```cpp
int gcd(int a, int b) {
    if (b == 0) return a;      // 边界条件：余数为0，除数为答案
    return gcd(b, a % b);      // 递推关系：gcd(a,b) = gcd(b, a%b)
}
```

太漂亮了——短短三行！而且这是**尾递归**——递归调用是最后一步，编译器可以优化成循环。

来模拟一下 gcd(48, 18)：

```
gcd(48, 18) → gcd(18, 12) → gcd(12, 6) → gcd(6, 0) → 返回 6
```

每次调用，第二个参数取第一个参数除以第二个的余数。当第二个参数变成 0 时，第一个参数就是答案。这是古希腊数学家欧几里得两千多年前就发现的算法——用递归表达只需要 3 行代码！

**例子二：倒序输出数字的每一位**（板书）：

```cpp
void printDigits(int n) {
    if (n < 10) {                  // 边界：只剩一位数
        cout << n << " ";
        return;
    }
    cout << n % 10 << " ";         // 先输出个位
    printDigits(n / 10);           // 再递归处理剩下的
}
// printDigits(1234) → 输出 "4 3 2 1 "
```

这里用 `n % 10` 取个位，`n / 10` 去掉个位。递归缩小规模，直到只剩一位数。

**例子三：爬楼梯问题**（这是斐波那契的另一个马甲）：

假设你爬楼梯，每次可以爬 1 阶或 2 阶。爬到第 n 阶有多少种方法？

分析：到达第 n 阶，要么从第 n-1 阶走 1 步过来，要么从 n-2 阶走 2 步过来。所以 **ways(n) = ways(n-1) + ways(n-2)**。

边界：ways(1) = 1（只能一步），ways(2) = 2（一步+一步 或 一步两阶）。

这就是斐波那契！只不过索引偏移了一点。

```cpp
int climbStairs(int n) {
    if (n == 1) return 1;
    if (n == 2) return 2;
    return climbStairs(n - 1) + climbStairs(n - 2);
}
```

看到了吗？**同一个递推公式可以描述完全不同的现实问题**——这就是数学抽象的威力。

好，最后留一个思考：所有你能用递归解决的问题，都能用循环解决吗？理论上是的——任何递归都可以转化为递推。但有些问题（如树的遍历、分治算法）用递归表达极其自然，强行改成循环反而难读。所以**学递归不是为了替代循环，而是多一种思考问题的角度。**

```
递归 vs 递推：

新例子：
```

## 📌 小结
递归和递推是同一枚硬币的两面——递归自顶向下，表达力强；递推自底向上，效率高。gcd 是完美的尾递归例子（3 行代码搞定千年前的算法）。爬楼梯问题展示了同一个递推公式可以解决不同的问题。核心心法：先用递归理解问题的"分拆"本质，需要时再优化为递推。
