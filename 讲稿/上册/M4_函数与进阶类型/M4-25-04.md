# 第25章 第4课时：生命周期全景图与最佳实践——变量从生到死

**时长**：15-20分钟 | **对应教材**：第25章 25.5-25.7节

## 🎯 本课时目标
- 画出全局变量、局部变量、static 局部变量的完整生命周期线
- 认清全局变量的三大风险，建立正确的使用观念
- 掌握"局部优先"的最佳实践，能自觉将变量放在最小范围内

## 📝 讲稿正文

好，今天我们把这章串起来，画出每种变量的"一生"，然后总结一套你在实际编码中应该遵守的变量使用原则。

先画三条生命线（板书）：

```
                   创建          使用                  销毁
全局变量：    ──●──────[──────────●──────────]────→ 程序结束

局部变量：              ●调用  [──使用──●]  返回
                      创建           销毁

static局部：  ──●──────[──────────●──────────]────→ 程序结束
             首次调用时创建               程序结束才销毁
```

**全局变量**：程序启动就出生，程序结束才死亡。全程在线，谁都能找它玩。像学校的操场——任何时候都开着。

**普通局部变量**：函数被调用时才出生（在栈上），函数返回立即死亡。生命周期短得就像暑假——还没玩够就没了。

**static 局部变量**：首次执行到定义它的那一行时出生，一直活到程序结束。像开学第一天办的学生卡——办一次，用六年。

来，用一个综合程序验证这三条线（板书）：

```cpp
#include <iostream>
using namespace std;

int globalVar = 1;                    // 全局：程序启动就活

void test() {
    int localVar = 2;                 // 局部：每次调用test时创建
    static int staticVar = 3;         // static：首次调用时创建，之后保留
    
    cout << "globalVar=" << globalVar
         << " localVar=" << localVar
         << " staticVar=" << staticVar << endl;
    
    globalVar++;   // 全局 +1
    localVar++;    // 局部 +1（但每次都从2开始）
    staticVar++;   // static +1（保留）
}

int main() {
    test();  // globalVar=1 localVar=2 staticVar=3
    test();  // globalVar=2 localVar=2 staticVar=4
    test();  // globalVar=3 localVar=2 staticVar=5
    return 0;
}
```

localVar 每次都从 2 开始——因为它每次都是新生的。globalVar 和 staticVar 一路递增——因为它们的值被保留了。

好，现在说正事。**全局变量为什么被嫌弃？**

三大风险，记住就行：

**第一：追责困难。** 全局变量谁都能改。出了 bug，你发现 `health` 变成了 -20——但程序里 10 个函数都可能改过它。你要逐个排查。——如果 `health` 是局部的，只有一个函数能改，秒定位。

**第二：函数不独立。** 一个依赖全局变量的函数不能单独复用。你拷贝这个函数到另一个项目——抱歉，得把全局变量也带上，还得确保它的初始值正确、没被其他函数偷偷改过。这就叫"耦合"——互相缠在一起，分不开。

**第三：命名冲突。** 多个文件里不能有同名的全局变量——链接器会报"重定义"错误。在大型项目里，几十上百个文件，命名冲突几乎是必然的。

不过，信息学竞赛里有例外——**大数组放全局是合理的**，因为局部大数组会撑爆栈空间（通常只有几 MB）。所以竞赛代码里 `int a[100005]` 放外面是标准写法。但请注意：这只是为了避免栈溢出，不是因为全局好。

**那么，最佳实践是什么？**

一句话：**变量的活动范围越小越好。**

具体决策流程（遇到一个新变量时自问）：
1. 这个变量只在**一个代码块**（if/for）里用？→ 放块里
2. 只在**一个函数**里用？→ 放函数里
3. 需要**跨函数共享**？→ 优先考虑通过参数传递（有必要时用引用），而不是设全局
4. 真的必须全局？→ 放到最后再考虑，加注释说明原因

来看对比（板书）：

```cpp
// ✅ 好——所有变量都在最小范围内
int main() {
    int n; cin >> n;
    int sum = 0;
    for (int i = 0; i < n; i++) {   // i 只在循环里用
        int val; cin >> val;         // val 只在循环里用
        sum += val;
    }
    cout << sum << endl;
    return 0;
}

// ❌ 差——能放局部的全拉成了全局
int n, sum, val;  // 统统全局！没必要
int main() {
    cin >> n;
    for (int i = 0; i < n; i++) { cin >> val; sum += val; }
    cout << sum << endl;
    return 0;
}
```

左边版本：`n`、`sum`、`val` 都是局部的——只存在于它们被需要的地方。出了作用域就自动销毁，不会干扰任何其他代码。右边版本：全部丢到全局——程序简单时看不出问题，复杂了就等着找 bug 找到崩溃。

从今天开始，养成一个习惯：**每个变量都问问自己——它能不能再小一圈？** 能放块里不放到函数里，能放函数里不放到全局。这是区分"新手代码"和"职业代码"的关键标志之一。

## 🖥️ 板书/屏幕要点
```
生命周期线：
  全局：程序启动 → 程序结束
  局部：函数调用 → 函数返回
  static局部：首次调用 → 程序结束

全局变量三大风险：
  1. 追责困难（谁都能改）
  2. 函数不独立（耦合）
  3. 命名冲突（多文件）

最佳实践：
  能放块里 → 不放函数里
  能放函数里 → 不放全局
  跨函数共享 → 优先用参数传递
```

## 📌 小结
三种变量的生命周期各不相同：全局和 static 从程序启动到结束，普通局部从函数调用到返回。全局变量有追责困难、函数不独立、命名冲突三大风险。最佳实践是"局部优先"——变量作用域越小越好，能放块里不放到函数里，能放函数里不放到全局。
