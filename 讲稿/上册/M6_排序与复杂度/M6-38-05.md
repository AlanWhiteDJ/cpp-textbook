## 📝 讲稿正文

前面四节课我们学了一整套复杂度分析的理论。今天我们来一次"综合实战"——把理论落地，亲自感受复杂度的威力。

首先，我教大家一个实用的小工具：**clock() 函数**。它可以测量程序运行了多长时间。用法很简单：

```cpp
#include <iostream>
#include <ctime>
using namespace std;

int main() {
    int n;
    cin >> n;
    clock_t start = clock();    // 记录开始时间
    
    // 你要测试的代码
    int s = 0;
    for (int i = 1; i <= n; i++)
        for (int j = 1; j <= n; j++)
            s++;
    
    clock_t end = clock();      // 记录结束时间
    double time = double(end - start) / CLOCKS_PER_SEC;
    cout << "执行时间: " << time << " 秒" << endl;
    return 0;
}
```

注意 `CLOCKS_PER_SEC` 是 clock 的"单位换算"——把 clock 的滴答数转成秒。

好，我们来做一个实验：写一个双重循环 O(n²)，分别测试 n=1000、2000、4000、8000 时的运行时间。你会发现：n 翻一倍，时间大约变成 4 倍！因为 (2n)² = 4n²。这就验证了 O(n²) 的增长规律。

再测一个单层循环 O(n)：n 翻一倍，时间大约翻一倍——符合 O(n)。

现在我们来解一道典型的"分析题"。看下面这段代码，分析它的复杂度和空间复杂度：

```cpp
const int MAX = 10005;
int sum[MAX][MAX];

int main() {
    int n;
    cin >> n;
    for (int i = 1; i <= n; i++)
        for (int j = 1; j <= n; j++)
            sum[i][j] = sum[i-1][j] + sum[i][j-1] - sum[i-1][j-1] + a[i][j];
}
```

时间上：双重循环，每层从 1 到 n，所以是 O(n²)。但注意这里 `sum` 数组是 MAX×MAX = 1 亿个 int，那就大约是 **400MB**！空间上直接超限了。所以这段代码在竞赛中是拿不到满分的——不是因为时间，而是因为内存。

这也是为什么要做复杂度分析：**在你写代码之前，就要预判它会不会超时、会不会超内存**。

再来一个综合判断。题目说：n ≤ 200000，每个数在 1 到 10⁹ 之间。问你能不能 O(n²) 排序？不能！n² 是 4×10¹⁰，肯定会超时。那 O(n log n) 呢？n log n 大约是 200000 × 18 ≈ 360 万，完全可行。所以结论：这道题得用 O(n log n) 的排序。

今天我们还要接触一个"求最大子段和"的经典问题来感受复杂度差异。方法 A 用三重循环 O(n³)，方法 B 用两重循环 O(n²)，方法 C 用一遍扫描 O(n)。分别测试 n=100、1000、5000，你会发现：方法 A 在 n=1000 时已经很吃力了，方法 B 在 n=5000 时还行，方法 C 在 n=100000 时依然飞快。

这就是复杂度的力量——**同样的功能，O(n) 和 O(n³) 的差距是天壤之别**。希望大家通过今天的实战，真正把复杂度分析变成你解题时的第一反应。

## 🖥️ 板书/屏幕要点
- **clock() 用法**：记录 `start` → 运行代码 → 记录 `end` → `(end-start)/CLOCKS_PER_SEC`
- **验证 O(n²)**：n 翻倍 → 时间约 4 倍
- **写代码前预判**：时间能否过？内存能否过？
- **最大子段和**：O(n³) vs O(n²) vs O(n) 的实际感受
